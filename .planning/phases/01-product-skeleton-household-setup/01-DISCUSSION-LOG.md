## 登录与首启方式

| Option | Description | Selected |
|--------|-------------|----------|
| 先免登录体验，等需要保存时再登录 | 降低首访门槛 | |
| 一进来就登录后再建档 | 先锁定用户和家庭归属 | Yes |
| 先做单机本地版，不做账号体系 | 最快，但偏离长期模型 | |

**User's choice:** 必须登录后使用  
**Notes:** 产品长期仍以账号化家庭数据为基础，不走免登录或纯本地模式。

| Option | Description | Selected |
|--------|-------------|----------|
| 一个家庭先只对应一个主账号 | MVP 简化协同复杂度 | Yes |
| 一个家庭从第一版就支持多人协同 | 提前覆盖共享场景 | |
| 先不显式建账号关系，后面再补 | 延后关系建模 | |

**User's choice:** 一个家庭先只对应一个主账号  
**Notes:** 多人协同明确不在 Phase 1 范围内。

| Option | Description | Selected |
|--------|-------------|----------|
| 手机号验证码登录 | 贴近中国大陆用户习惯 | Yes |
| 邮箱验证码登录 | 实现相对简单 | |
| 邮箱 + 密码登录 | 通用但不贴近目标市场 | |

**User's choice:** 长期方向为手机号验证码登录  
**Notes:** 这是长期方向，不要求在 Phase 1 接真实短信能力。

| Option | Description | Selected |
|--------|-------------|----------|
| 登录成功后直接进入家庭建档向导 | 主流程最短 | Yes |
| 先到一个空首页，再引导去建档 | 有首页承接感 | |
| 先到设置页，用户自己开始建档 | 更偏后台式流程 | |

**User's choice:** 登录后直接进入家庭建档  
**Notes:** 用户希望首版尽快进入产品主流程。

| Option | Description | Selected |
|--------|-------------|----------|
| 手机号 + 密码登录 | 用数据库预置测试账号即可跑通 | Yes |
| 手机号 + 固定验证码登录 | 更像未来形态 | |
| 仅开发模式自动登录一个测试账号 | 最快，但产品感最弱 | |

**User's choice:** 先做数据库预置测试账号的 `手机号 + 密码登录`  
**Notes:** 用户特别说明第一版登录不先做真实能力，可以手动创建账户。

## 家庭建档流程

| Option | Description | Selected |
|--------|-------------|----------|
| 单页表单，一次填完 | 最直接、实现最简单 | Yes |
| 分步向导，3 到 4 步完成 | 更有引导感 | |
| 极简首填，只填最少字段，其他后补 | 降低阻力 | |

**User's choice:** 单页表单，一次填完  
**Notes:** 当前阶段优先简单可维护，不做多步向导。

| Option | Description | Selected |
|--------|-------------|----------|
| 只保留最关键字段 | 最符合 MVP 冷启动目标 | Yes |
| 在关键字段基础上，把常购平台、提醒容忍度也设为必填 | 信息更完整 | |
| 更极简，只填家庭人数、宠物/婴幼儿情况 | 更快，但信息不足 | |

**User's choice:** 只保留关键字段  
**Notes:** 必填字段锁定为家庭人数、宠物/婴幼儿情况、储物空间等级、价格敏感度、囤货偏好。

| Option | Description | Selected |
|--------|-------------|----------|
| 直接进入“添加追踪物品”流程 | 主流程不断开 | Yes |
| 先进入空首页，再提示去添加物品 | 有首页承接 | |
| 先进入设置页，用户自行继续 | 偏后台式操作 | |

**User's choice:** 建档完成后直接进入添加追踪物品  
**Notes:** 用户希望主流程尽量短。

## 物品添加方式

| Option | Description | Selected |
|--------|-------------|----------|
| 先从预设支持品类里选择，再补充品牌/规格 | 数据质量更稳 | Yes |
| 完全自由输入 | 最自由 | |
| 两者并存，默认先选预设，也允许自定义 | 灵活性更高 | |

**User's choice:** 先从预设支持品类里选择，再补品牌和规格  
**Notes:** 主流程先走预设支持品类。

| Option | Description | Selected |
|--------|-------------|----------|
| 不允许，只能选支持品类 | 最有利于控制数据质量 | |
| 允许，但明确标记“暂不支持价格追踪” | 允许自由度但显式限制 | |
| 允许，而且和标准品类完全同等待遇 | 产品层不做差别待遇 | Yes |

**User's choice:** 允许自定义物品，且和标准品类同等待遇  
**Notes:** 这里已明确提醒：产品规则可平等，但 Phase 3 的真实抓价能力仍受价格源约束。

| Option | Description | Selected |
|--------|-------------|----------|
| 创建时只填最少字段：品类、名称/别名、单位 | 最轻量 | |
| 创建时把品牌、常用规格、安全库存天数、首选平台都填全 | 信息最完整 | |
| 折中：创建时填品类、品牌、规格，其他后补 | 平衡冷启动与数据质量 | Yes |

**User's choice:** 创建时填品类、品牌、规格，其他后补  
**Notes:** 这是 Phase 1 的物品创建字段范围。

## Phase 1 的产品壳范围

| Option | Description | Selected |
|--------|-------------|----------|
| 不做，先只做登录 / 建档 / 物品管理几个独立页面 | 最省实现量 | |
| 要做，先把未来的一级导航壳搭出来 | 提前建立产品骨架 | Yes |

**User's choice:** Phase 1 要把一级导航壳一起做出来  
**Notes:** 用户希望 Phase 1 就立起产品基础外壳。

| Option | Description | Selected |
|--------|-------------|----------|
| 首页 / 物品 / 记录 / 设置 | 带占位首页和记录入口 | |
| 首页 / 物品 / 记录 / 优惠 / 设置 | 更接近完整产品 | |
| 只保留 物品 / 设置 | 最收敛 | Yes |

**User's choice:** 首版只保留 `物品 / 设置`  
**Notes:** 这一选择与“做首页壳”产生冲突，后续通过补问解决。

| Option | Description | Selected |
|--------|-------------|----------|
| 只做一个占位空首页，用于承接后续 Phase 4 | 先留位置 | |
| 做一个基础首页，显示“先完成建档和添加物品”的引导 | 有引导感 | |
| Phase 1 不做首页，建档后直接进物品页 | 与导航收敛一致 | Yes |

**User's choice:** 最终改为不做首页，建档后直接进入物品页  
**Notes:** 用户在冲突澄清后选择 `C`，因此首页不进入 Phase 1。
