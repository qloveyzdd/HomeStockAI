## 记录入口与页面结构

| Option | Description | Selected |
|--------|-------------|----------|
| 从物品列表和物品详情进入，不单独给一级导航入口 | 延续 Phase 1 的主工作区和最小导航 | Yes |
| 新增一级导航 `记录` | 把记录作为新的主入口 | |
| 只从设置页或隐藏入口进入 | 保持入口最少，但可发现性差 | |

**User's choice:** 购买记录主入口放在物品列表和物品详情，不新增一级导航。  
**Notes:** Phase 2 继续围绕 `物品` 做主工作区，而不是扩产品壳。

| Option | Description | Selected |
|--------|-------------|----------|
| 做独立物品详情页，再从详情进入编辑 | 让详情承接记录和库存状态 | Yes |
| 继续直接在编辑页里承载记录相关能力 | 少一个页面，但语义更混杂 | |
| 先不做详情，只做独立新增记录页 | 结构简单，但会削弱物品视角 | |

**User's choice:** 要做独立物品详情页。  
**Notes:** 现有编辑页后续应逐步转成“详情页 -> 再进编辑”的结构。

| Option | Description | Selected |
|--------|-------------|----------|
| 继续以 `物品` 页为主工作区 | 与 Phase 1 保持一致 | Yes |
| 改成 `记录` 页 | 让录入行为变成主任务 | |
| 改成新首页 | 恢复 dashboard 方向 | |

**User's choice:** 主工作区继续放在 `物品` 页。  
**Notes:** 不恢复首页，也不切换一级导航重心。

## 手动录入表单

| Option | Description | Selected |
|--------|-------------|----------|
| 从物品详情页点“添加购买记录”，默认绑定当前物品 | 最符合补货决策语境 | Yes |
| 做全局“添加记录”页，进去后再选物品 | 更通用，但多一步选择 | |
| 两种都做 | 更灵活，但超出 MVP 需要 | |

**User's choice:** 手动录入从物品详情页进入，默认绑定当前物品。  
**Notes:** 录入动作优先服务当前物品，而不是全局流水视角。

| Option | Description | Selected |
|--------|-------------|----------|
| 物品、购买数量、总价、购买日期、平台 | 最小可用字段集 | Yes |
| A + 订单标题必填 | 数据更全，但增加录入负担 | |
| A + 到货日期必填 | 对后续精细化更有帮助，但当前过重 | |

**User's choice:** 必填字段锁定为物品、购买数量、总价、购买日期、平台。  
**Notes:** 其他字段都不能变成 Phase 2 的录入门槛。

| Option | Description | Selected |
|--------|-------------|----------|
| 预设平台选项 + 允许自定义 | 兼顾规范化和灵活性 | Yes |
| 完全自由输入 | 最自由，但不利于后续统计 | |
| 只允许预设平台 | 最规整，但约束过强 | |

**User's choice:** 平台字段采用预设 + 自定义。  
**Notes:** 平台标准化要做，但不能把用户卡死。

| Option | Description | Selected |
|--------|-------------|----------|
| 一次只录一条，保存后可继续添加 | 最简单、最稳 | Yes |
| 一次可连续录多条同物品记录 | 更快，但实现更重 | |
| 做表格式批量录入 | 偏后台，不符合当前产品感 | |

**User's choice:** 一次只录一条。  
**Notes:** 先把单条录入闭环做顺，再考虑批量。

## 订单文本粘贴解析

| Option | Description | Selected |
|--------|-------------|----------|
| 放在“添加购买记录”页里，作为次级入口 | 不扩新页面，最符合 MVP | Yes |
| 单独做“导入记录”页 | 能力更独立，但增加壳层复杂度 | |
| 先不做文本解析入口 | 范围更小，但少了冷启动降阻手段 | |

**User's choice:** 文本解析入口放在“添加购买记录”页里。  
**Notes:** 它是手动录入的辅助能力，不是新主流程。

| Option | Description | Selected |
|--------|-------------|----------|
| 解析出一条候选记录，落到表单里让用户确认后保存 | 最稳、最容易解释 | Yes |
| 解析出多条候选记录列表，用户逐条勾选确认 | 更强，但更重 | |
| 只做字段辅助提取，不自动映射成表单 | 更轻，但价值感偏弱 | |

**User's choice:** 先解析单条候选记录，再回填表单确认。  
**Notes:** 用户要的是减负，不是黑盒自动导入。

| Option | Description | Selected |
|--------|-------------|----------|
| 只追求物品名/数量/价格/平台的基础能力 | 符合 Phase 2 的“够用”目标 | Yes |
| 尽量把规格、优惠信息、到货时间也一起解析 | 更强，但明显增重 | |
| 只解析商品名，其他都手填 | 价值偏低 | |

**User's choice:** 只覆盖基础字段提取。  
**Notes:** Phase 2 不追求复杂订单理解。

## 库存估算与低置信度展示

| Option | Description | Selected |
|--------|-------------|----------|
| 要，给粗略估算，但明确标记“估算中” | 最符合“先验证有用”目标 | Yes |
| 不要，记录不足就不显示估算结果 | 更保守，但验证价值不足 | |
| 只显示是否低于安全库存，不显示剩余天数 | 太弱，无法形成判断感 | |

**User's choice:** 记录不足时也先给粗略估算，但必须标记“估算中”。  
**Notes:** 早期不准可以接受，但不能假装确定。

| Option | Description | Selected |
|--------|-------------|----------|
| 优先用最近 2 到 5 条记录；不足时退化到最近 1 条 + 家庭档案/品类默认值 | 平衡可信度和冷启动可用性 | Yes |
| 必须至少有 2 条记录才开始估算 | 更稳，但过于保守 | |
| 只用最近 1 条记录，先不做多记录平均 | 最快，但容易失真 | |

**User's choice:** 用 2 到 5 条记录为主，不足时退化。  
**Notes:** 这是 Phase 2 最关键的估算口径。

| Option | Description | Selected |
|--------|-------------|----------|
| 先展示在物品详情页，同时在列表页给简化状态 | 详情完整、列表轻量 | Yes |
| 只展示在详情页 | 更克制，但列表没有反馈 | |
| 直接在列表页完整展示 | 信息太重 | |

**User's choice:** 详情页完整展示，列表页给简化状态。  
**Notes:** 继续保持 `物品` 页可扫读、详情页可解释的结构。

| Option | Description | Selected |
|--------|-------------|----------|
| 只显示“预计剩余天数/估算中/低于安全库存”等轻量信息 | 最适合列表密度 | Yes |
| 把日均消耗、置信度、最近记录时间都堆上去 | 信息太满 | |
| 暂时不在列表页展示库存状态 | 少了一半价值 | |

**User's choice:** 列表页只显示轻量库存状态。  
**Notes:** 复杂解释只留在详情页。
