# Architecture Research: 家补 AI（RefillWise）

## 设计原则

这个项目最本质的系统结构不是商城，也不是记账工具，而是一个“以家庭消耗品为核心的数据判断系统”。所以架构应该服务于一条清晰主线：

记录事实 -> 推断库存 -> 采集价格 -> 生成建议 -> 接收反馈 -> 再次修正

## 推荐结构

### 1. Web App

职责：

- 首次建档
- 物品管理
- 购买记录录入
- 首页决策展示
- 详情解释与反馈

建议：

- `apps/web`
- 只负责 UI、交互、表单校验和展示逻辑
- 不在前端做真正的库存与推荐计算

### 2. API App

职责：

- Household / Item / PurchaseRecord CRUD
- InventoryState 计算接口
- PriceSnapshot 查询接口
- Recommendation 查询接口
- FeedbackLog 写入

建议：

- `apps/api`
- 按领域模块组织，不按技术层堆文件
- 一个后端应用里先包含所有业务模块

### 3. Worker

职责：

- 定时库存重算
- 定时抓价
- 抓价后重算推荐
- 通知节流和触发

建议：

- `apps/worker`
- 和 API 共用同一套 domain/service 层

### 4. Storage

职责：

- `PostgreSQL` 存结构化业务数据
- `Redis` 存任务、节流键、短期缓存

## 推荐的后端模块边界

| 模块 | 主要职责 | 是否独立部署 |
|------|----------|--------------|
| household | 家庭档案、偏好、提醒容忍度 | 否 |
| items | 追踪物品、支持品类、安全库存配置 | 否 |
| records | 购买记录录入、订单文本解析 | 否 |
| inventory | 消耗速度估算、剩余天数计算、置信度 | 否 |
| pricing | 价格抓取、价格快照、历史对比指标 | 否 |
| recommendation | 动作判断、推荐数量、原因解释 | 否 |
| notification | 去重规则、发送策略、免打扰 | 否 |

MVP 阶段这些模块都应该是同一个后端仓内的领域模块，不拆服务。

## 核心数据流

### 首次使用

1. 用户创建家庭档案
2. 用户添加追踪物品
3. 用户录入 1 到 3 条购买记录
4. API 触发库存初算
5. 如已有价格数据，则同步生成第一版建议
6. 首页展示“快见底 / 值得买 / 先别买”

### 日常运行

1. 定时任务抓取价格
2. 写入 `PriceSnapshot`
3. 触发建议重算
4. 如命中提醒条件，再进入通知节流判断
5. 用户点击反馈
6. 写入 `FeedbackLog`
7. 触发库存与建议重新计算

## 同步 API 与异步任务的分工

### 同步 API 负责

- 用户可感知的即时操作
- 表单提交后的即时刷新
- 查询当前状态

### 异步任务负责

- 批量库存重算
- 定时抓价
- 批量推荐刷新
- 通知发送和去重

规则很简单：凡是可以等几秒、几分钟、或定时运行的，都不要塞进用户请求链路。

## MVP 最佳建设顺序

1. **基础骨架**
   - 前后端目录
   - 数据库连接
   - Household / Item 基础模型
2. **记录与库存**
   - PurchaseRecord
   - InventoryState
   - 基础重算逻辑
3. **首页最小闭环**
   - 首页卡片
   - 物品列表
   - 详情页
4. **价格链路**
   - PriceSnapshot
   - 单一抓价源
   - 简单历史指标
5. **推荐与反馈**
   - Recommendation
   - FeedbackLog
   - 反馈后重算
6. **提醒规则**
   - 去重
   - 稍后提醒
   - 不再提醒

## 现在不要做的架构动作

- 不拆微服务
- 不引入事件总线
- 不做读写分离
- 不建单独分析仓
- 不为未来 App 先抽“超级通用 BFF”

## 一句话总结

这个项目的正确架构不是“很多服务”，而是“一个前端、一个后端、一个 worker，共享同一套领域逻辑，把同步体验和异步计算分开”。
