# Feature Research: 家补 AI（RefillWise）

## 判断标准

这个产品不是“家庭物品都能管”，也不是“优惠越多越好”，而是“在家庭补货场景里给出有库存语境的购买结论”。所以功能分层要围绕这个判断。

## Table Stakes

这些不是差异化，但没有就无法形成最小闭环。

| 功能 | 复杂度 | 依赖 | 说明 |
|------|--------|------|------|
| 家庭档案 | 低 | 无 | 用来初始化消耗画像、提醒强度和囤货偏好 |
| 追踪物品管理 | 低 | 家庭档案 | 用户必须能明确“我到底在管哪些东西” |
| 购买记录录入 | 中 | 追踪物品 | 没有购买记录，库存估算就没有起点 |
| 库存余量 / 剩余天数估算 | 中 | 购买记录、家庭档案 | 这是所有决策的底座 |
| 安全库存判断 | 低 | 库存估算 | 用来决定什么叫“快没了” |
| 价格快照与历史对比 | 中 | 追踪物品 | 必须至少知道现在贵不贵 |
| 首页决策视图 | 中 | 推荐引擎 | 用户打开就要看到“现在该做什么” |
| 用户反馈校正 | 中 | 库存估算、推荐引擎 | 冷启动阶段最重要的学习机制 |
| 提醒去重规则 | 中 | 推荐引擎、反馈 | 没有克制的提醒会直接劝退用户 |

## Differentiators

这些是让产品真正不像“库存表格”或“导购工具”的关键。

| 功能 | 复杂度 | 依赖 | 为什么重要 |
|------|--------|------|------------|
| 有库存语境的“值得买”判断 | 中 | 库存估算、价格快照 | 不是便宜就推，而是结合家里还剩多少来判断 |
| 推荐购买数量 | 中 | 囤货偏好、储物空间、价格机会 | 建议不是“买不买”，而是“现在买几件更合理” |
| 建议解释链路 | 中 | 推荐引擎 | 用户要知道系统为什么这么判断，才能建立信任 |
| 家庭偏好驱动的提醒节奏 | 中 | 家庭档案、反馈 | 同样是低价，有人想囤，有人不想被打扰 |
| 反馈驱动的持续校正 | 中 | FeedbackLog | 这是产品从“猜”变成“越来越懂你”的关键 |

## Anti-Features

这些功能看起来相关，但会把 MVP 迅速带偏。

| 功能 | 为什么现在不做 |
|------|----------------|
| 全屋库存可视化大盘 | 会把产品拉成家庭 ERP，稀释补货决策核心 |
| 全网促销瀑布流 | 会让产品被误解成导购工具 |
| 自动下单 | 需要极高信任度和交易整合成本 |
| 生鲜 / 零食 / 调味料复杂预测 | 波动太大，冷启动误差很高 |
| 多成员家庭协作 | 首版先把单人闭环跑顺 |
| 社交、PK、薅羊毛社区 | 与核心价值关系弱，容易制造噪音 |

## MVP 应该怎么收

### v1 必须做到

- 用户能快速建档并添加少量核心物品
- 用户能录入购买记录并立刻看到库存变化
- 系统能稳定输出三类关键视图：
  - 快见底
  - 现在值得买
  - 先别买
- 用户能用极低成本纠正系统

### v1 明确不追求

- 抓很多平台
- 推荐很多 SKU
- 复杂个性化模型
- 精确到“家里还剩多少张纸”

## 功能依赖顺序

1. 家庭档案
2. 追踪物品
3. 购买记录
4. 库存估算
5. 价格快照
6. 推荐引擎
7. 首页决策视图
8. 反馈校正
9. 提醒规则

## 一句话总结

MVP 最值钱的功能不是“多”，而是“让用户在 3 秒内知道现在该不该买，并且愿意顺手纠正系统”。
