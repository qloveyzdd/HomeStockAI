# Pitfalls Research: 家补 AI（RefillWise）

## 关键坑点

| 坑点 | 预警信号 | 预防方式 | 适合处理阶段 |
|------|----------|----------|--------------|
| 冷启动录入过重 | 新用户在建档或首批录入时流失，3 个物品都没加完 | 首次只要求少量核心物品和少量记录，优先支持手动录入 + 文本粘贴 | Phase 1-2 |
| 把“估算”伪装成“精确” | 用户经常吐槽“根本不准”，但界面仍显示过度确定的数字 | 所有库存结果都显示置信度，低置信度直接标“估算中” | Phase 2 |
| SKU 规格不统一导致价格误判 | 同一物品不同规格被当成同价比较，推荐经常失真 | 早期先约束支持的规格和追踪数量，所有价格都落到单位价格层 | Phase 3 |
| 抓价源不稳定拖垮主链路 | 首页常出现价格空白或任务失败堆积 | 抓价与推荐分离；抓价失败时库存与补货建议仍可工作 | Phase 3 |
| 首页变成优惠流 | 用户只看到“促销”，看不懂“为什么对我有用” | 首页模块始终按决策分类，不按平台优惠瀑布流组织 | Phase 4 |
| 推荐黑盒化 | 用户不知道为什么建议买/不买，逐渐不信任 | 每条建议都必须附原因说明，原因要能落到库存、价格、偏好三个维度 | Phase 4 |
| 反馈没有真正反哺模型 | 用户点了“家里还有”后，下次还是同样提醒 | 每种反馈都必须映射到库存、速度、提醒节奏的可见变化 | Phase 5 |
| 提醒过多 | 打开率下降，mute 比例快速上升 | 强制去重、低价值不提醒、同物品 3 天内最多 1 次 | Phase 5 |
| 把价格机会看得比缺货风险更重 | 系统总在“便宜”时推送，但用户真正痛点是别断货 | 决策引擎里缺货风险优先级要高于 deal score | Phase 4 |
| 没有可回溯的决策依据 | 线上出了错误建议，但无法定位是库存、价格还是反馈导致 | Recommendation 需要保留关键计算字段和 reason_text | Phase 4 |

## 额外提醒

### 1. 不要过早做“智能感”

这个产品最容易犯的错是急着做 OCR、LLM、自动化推荐文案，看起来很聪明，但底层库存和价格判断还不稳。顺序必须反过来：先把事实层和规则层做好，再谈“智能”。

### 2. 不要为了可扩展把数据库设计抽太虚

MVP 只需要把家庭、物品、购买记录、价格快照、库存状态、建议、反馈七类对象跑顺。过早做泛化商品模型、跨平台 SKU 抽象，会拖慢速度。

### 3. 不要把通知当增长工具

这个产品里的通知不是为了刷打开率，而是为了在真正关键时刻节省决策成本。通知多了，信任就没了。

## 一句话总结

这个方向最大的风险不是功能不够多，而是系统在“还没准”的时候装得太准、在“没必要提醒”的时候提醒得太勤。
